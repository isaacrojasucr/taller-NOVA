1. DECISION DE DISENO: HARNESS LIGERO PROPIO VS. iRPGUnit
-----------------------------------------------------------------
El punto 6.4/15.4 de Revision_IBMi.md exige pruebas unitarias de las
reglas clave de conciliacion y validacion estructural del JSON. El
framework iRPGUnit (de uso comun en proyectos IBM i) requiere prototipos
y convenciones externas (RPGUNIT.RUTSTCASE, anotaciones //TEST, etc.) que
no es posible verificar con certeza sin un compilador IBM i disponible en
este entorno; un nombre o firma incorrecto produciria modulos que no
compilan.

Para no asumir ese riesgo, este taller implementa un harness de pruebas
autocontenido y deliberado:

  - Cada modulo de prueba (TSTBALCA, TSTCONCI, TSTJSONB) declara
    localmente los prototipos EXTPROC de los procedimientos bajo prueba
    (de GLBALCAL/GLCONCIL/GLJSONBL respectivamente) - sin depender de
    copy members externos.
  - Cada modulo exporta UN UNICO procedimiento "driver":
      TSTBCRUN(pDetalle: varchar(2000)): int(10)   -- TSTBALCA
      TSTCCRUN(pDetalle: varchar(2000)): int(10)   -- TSTCONCI
      TSTJBRUN(pDetalle: varchar(2000)): int(10)   -- TSTJSONB
  - El driver ejecuta varios casos (TSTBC01.., TSTCC01.., TSTJB01..),
    cada uno con un assert manual (IF/ELSE) que agrega una linea
    "<CASO> OK: <descripcion>" o "<CASO> FALLO: <valores obtenidos>"
    (separadas por x'0A') a pDetalle, e incrementa un contador de fallos.
  - El driver retorna la cantidad total de casos fallidos (0 = todo OK).

Esta decision se documenta explicitamente como un tradeoff: se prioriza
tener pruebas EJECUTABLES Y COMPILABLES con certeza (sin dependencias
externas no verificables) sobre el uso de un framework estandar de la
comunidad. El patron (driver + reporte de texto + contador de fallos) es
facilmente migrable a iRPGUnit en el futuro si el entorno objetivo ya
cuenta con esa libreria instalada: cada IF/ELSE de assert manual se
traduce a una llamada ASSERTEQUAL/ASSERTTRUE de iRPGUnit dentro de un
procedimiento //TEST.

2. COMO EJECUTAR LAS PRUEBAS
--------------------------------
TSTSRV (UNITSRCSRV/TSTSRV.bnd) enlaza TSTBALCA, TSTCONCI y TSTJSONB
(ver DEPLOY.txt, Fase G). Para ejecutar todas las pruebas, crear un
pequeno programa CL o RPG que invoque los 3 drivers y muestre el
resultado, por ejemplo:

  dcl-s wDet1 varchar(2000);
  dcl-s wDet2 varchar(2000);
  dcl-s wDet3 varchar(2000);
  dcl-s wFallos int(10);

  wFallos = TSTBCRUN(wDet1) + TSTCCRUN(wDet2) + TSTJBRUN(wDet3);
  // mostrar wDet1, wDet2, wDet3 (cada linea separada por x'0A')
  // y wFallos (0 = todas las pruebas pasaron)

    GLCONTINI();
    GLCONTOT('CONCILIADA', saldoFte=1000, saldoConc=1000, difNeta=0);
    GLCONTOT('NO_CONCILIADA', saldoFte=2000, saldoConc=1800, difNeta=200);
    GLCONTGET(...)

              sumFte=3000, sumConc=2800, sumDif=200

5. CASOS DE PRUEBA - TSTJSONB (GLJSONBL)
------------------------------------------------
  TSTJB01 - GLJSNMETA('QA') incluye "ambiente":"QA" y
            "versionEstructura":"1.0.0"

  TSTJB02 - GLJSNCTX(...) incluye el bloque anidado
            "periodo":{"anio":2026,"mes":6,...} en el orden correcto

  TSTJB03 - GLJSNINC con un mensaje que contiene comillas dobles (")
            produce el caracter escapado \" en el JSON resultante
            (GLJSNESC)

  TSTJB04 - GLJSNCTA con excedeTolerancia=*on y requiereRevision=*off
            produce "excedeTolerancia":true y "requiereRevision":false
            (GLJSNBOOL)

  TSTJB05 - GLJSNINI() reinicia el acumulador de cuentas; dos llamadas a
            GLJSNCTA seguidas de GLJSNCCNT()=2 y GLJSNCGET(3)="" (fuera
            de rango devuelve vacio)

6. VALIDACION ESTRUCTURAL DEL JSON DE SALIDA (GLCNC001)
---------------------------------------------------------------
El JSON publicado en pRutaJson debe cumplir, como minimo:

  - Es un objeto JSON valido (parseable) con las claves de primer nivel,
    en este orden: metadata, ejecucion, contexto, cuentas, controlTotales,
    incidentes.
  - "metadata".versionEstructura = "1.0.0", "ambiente" coincide con el
    parametro pAmbiente.
  - "ejecucion".idEjecucion sigue el patron GLCNC<timestamp ISO sin '-'/
    '.'> y coincide con el idEjecucion de la bitacora (BITACORA.txt).
  - "ejecucion".estadoEjecucion en {FINALIZADO, PARCIAL, ERROR}
    (ver ESTADOSC.txt) y es consistente con la severidad maxima de
    "incidentes".
  - "contexto".periodo.anio/mes coinciden con UTLANIO/UTLMES(pFchProc).
  - "cuentas" es un arreglo; cada elemento tiene cuentaContable dentro
    del rango [pCtaIni, pCtaFin] y codigoBanco/codigoSucursal/
    codigoMoneda = pBanco/pSucursal/pMoneda.
  - Cada cuenta cuyo estadoConciliacion.codigo sea NO_CONCILIADA o
    PARCIAL tiene un incidente correspondiente en "incidentes" con la
    misma cuentaContable (codigo ERR-GL-001/WARN-GL-001 respectivamente).
  - "controlTotales".totalCuentasProcesadas = numero de elementos en
    "cuentas"; totalCuentasConciliadas = cantidad de elementos con
    estadoConciliacion.codigo = "CONCILIADA"; totalCuentasConDiferencia =
    cantidad de elementos con diferencias.diferenciaNeta <> 0.
  - "controlTotales".sumatoriaSaldoFinalFuente/Conciliado/
    sumatoriaDiferenciaNeta son la suma de los campos homonimos de
    "cuentas[].saldos"/"cuentas[].diferencias".
  - Cada elemento de "cuentas" con cantidadMovimientos=0 tiene
    primerMovimiento=null y ultimoMovimiento=null (no ausentes, no
    cadena vacia).
