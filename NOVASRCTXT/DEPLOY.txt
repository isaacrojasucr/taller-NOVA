FASE A - BASE DE DATOS (115 objetos SQL, NOVASRCSQL/)
----------------------------------------------------------
Ejecutar cada .sql con RUNSQLSTM (o STRSQL / Run SQL Scripts) en el orden
indicado, respetando las dependencias de FK (CONSTRAINT). Todos los
objetos son CREATE OR REPLACE TABLE/VIEW; volver a ejecutar un script no
falla si el objeto ya existe.

  RUNSQLSTM SRCSTMF('/ruta/NOVASRCSQL/<archivo>.sql') COMMIT(*NONE)

5. DEPENDENCIA: HASH_SHA256 (GLJSNHASH)
-------------------------------------------
GLJSNHASH (modulo GLJSONBL) usa la funcion SQL HASH_SHA256 sobre
SYSIBM.SYSDUMMY1 para calcular "trazabilidad.hashCuenta". HASH_SHA256
esta disponible en Db2 for i a partir de cierto nivel de PTF (funciones
de hashing del grupo de PTF de base de datos). Si el SQLSTT al compilar o
ejecutar GLJSONBL indica funcion no encontrada (SQL0204 sobre
HASH_SHA256), verificar el nivel de PTF de Db2 for i del LPAR de PUB400.
Como mitigacion documentada (sin cambiar la firma de GLJSNHASH ni el
contrato GLOUTSRV), una alternativa seria sustituir HASH_SHA256 por
HASH_MD5 o por una concatenacion determinista sin hash; no aplicado en
este taller porque HASH_SHA256 esta disponible en el LPAR de pruebas.

6. DECISION DE DISENO: nivelCuenta COMO STRING EN EL JSON
---------------------------------------------------------------
GLMST.NIVEL_CUENTA se define como VARCHAR(50) (no numerico), porque
representa el nivel jerarquico de la cuenta dentro del plan de cuentas
como un codigo/etiqueta (p.ej. "Mayor", "Submayor", "Detalle" o una
nomenclatura jerarquica con puntos/prefijos), no una cantidad sobre la
que se realicen operaciones aritmeticas. En consecuencia,
"cuentaMayor.nivelCuenta" se transporta en el JSON como string (igual que
codigoBanco/codigoSucursal/cuentaContable, que tambien son
identificadores con apariencia numerica pero se tratan como codigos), lo
que evita perdida de ceros a la izquierda y problemas de formato/locale si
en el futuro el catalogo de niveles cambia de "1"/"2"/"3" a una
nomenclatura alfanumerica.
